## 2. The Unified Architecture (The Four Pillars)

The project evolved from a single AP workflow request into a comprehensive, four-pillar Microsoft 365-native architecture.

### Pillar 1: Automated Financial Allocation (AFA) Engine
*Formerly: AP Workflow*
- **Purpose:** Automates a complex, 4-pass AP payment allocation algorithm against dynamic budget constraints.
- **Mechanics:** User inputs budget → reviews/adjusts vendor priority → Python Azure Function executes deterministic allocation → user approves via Power Apps → final schedule is exported and bank ACH files are reconciled.
- **Key Requirement:** The algorithm *must* remain in Python to ensure deterministic fidelity; no-code tools cannot handle the looping/reallocation logic.

### Pillar 2: Executive Communications Intelligence
*Formerly: Executive Intelligence System*
- **Purpose:** AI-driven layer on top of Outlook to triage email, draft responses, and extract deal signals.
- **Mechanics:** Azure OpenAI categorizes inbound email by urgency, drafts context-aware replies, and detects deal-stage keywords to suggest calendar actions.

### Pillar 3: Universal Data Governance & Document AI
*Formerly: Universal Truth Architecture*
- **Purpose:** A structured file hierarchy (`01_CORPORATE`, `02_OPERATIONS`, `03_DEALS`, `04_GOVERNANCE`) powered by AI document recognition.
- **Mechanics:** All documents enter a `00_STAGING` folder. SharePoint Premium (Syntex) + AI Builder classifies the document, extracts metadata, recommends a standardized file name, and suggests a permanent filing location.

### Pillar 4: Employee Financial & Expense Hub
*Formerly: Personal Financial Tracker (McQuire Tracker)*
- **Purpose:** A secure tool for managing reimbursable expenses and personal financial tracking.
- **Mechanics:** Connects to personal bank accounts via Plaid. Uses a deterministic rules engine to classify transactions as Personal, LLC, or Peak 10.
- **The "Chinese Wall":** This is the most critical architectural constraint. Personal data lives in an encrypted Azure SQL partition. The classification engine runs *inside* the partition. Only explicitly approved "Expense Claim Records" (vendor, date, amount, receipt) are pushed across the boundary to the Peak 10 AP queue. The company never sees raw bank data.

## 3. Inter-Pillar Integrations

The system is designed to act as a single organism, not four separate apps:
1. **Outlook → Financial Hub:** Pillar 2 detects receipts (Uber, airlines) in email, extracts them, and pushes them to Pillar 4, which automatically attaches them to the corresponding Plaid bank transaction.
2. **Financial Hub → AFA Engine:** Approved expenses from Pillar 4 are pushed directly into the Pillar 1 AP invoice queue.
3. **Outlook → Document AI:** Pillar 2 detects executed contracts in email attachments and routes them directly to the Pillar 3 `00_STAGING` folder.
4. **AFA Engine → Document AI:** Final payment schedules generated by Pillar 1 are auto-named and filed by Pillar 3.

---

## 4. Continuous Learning & Process Intelligence

The user explicitly requested that the system be able to learn, adapt, and identify choke points.
- **The Signals Loop:** Every time the user corrects an AI recommendation (editing an email draft, changing a document filing location), the delta is logged. Models are periodically fine-tuned on these corrections.
- **Process Mining:** Power Automate Process Mining ingests event logs to identify bottlenecks (e.g., "Feedback requests to Contractor X stall the AP run by 3.5 days") and surfaces recommendations to the executive team.
